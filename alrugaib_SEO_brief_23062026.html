Agency Action Brief — Performance Investigation

Al Rugaib Furniture · Shopify (Expanse OS 2.0) · All Mobile-First
Prepared by: CRO Director Issued: June 2026 Scope: Investigation + Estimate only — no fix work until report received
Context for agency: All 13 tested pages fail Core Web Vitals on both mobile and desktop. Mobile performance scores range from 26–44. Mobile LCP is 3.3–3.9s across every page type. The site carries 96 installed Shopify apps. The primary objective of this engagement is to identify the root causes — specifically which scripts, apps, and assets are responsible — and to produce a written findings report with a time and cost estimate. No fix work should begin until the estimate is approved.
Agency Action Points
1

Run Third-Party Script Waterfall Audit

Record every external JS file loading on the homepage, one collection page (Dining Room), and one PDP (Hardt Beige HC03540). Measure each script's transfer size, parse time, and execution time individually.

  • Use WebPageTest (webpagetest.org) filmstrip + waterfall view, or Chrome DevTools Performance panel
  • Test on a throttled mobile profile: 4G (Motorola G4 or equivalent) — this reflects the majority Saudi user
  • Record: domain, file name, transfer size (KB), execution time (ms), load trigger (parser/async/defer/render-blocking)
  • Flag any script with execution time >50ms or transfer size >100 KiB as a candidate for deferral or removal
Deliverable: Waterfall screenshot + table of all third-party scripts with size and execution time per page type
Critical Week 1
2

Identify Which Apps Inject Global JavaScript — App List Provided Below

The site has 96 installed Shopify apps. Many inject JS on every page regardless of whether their functionality is needed on that page. This is the most likely single cause of the 2,400–3,500 KiB unused JS flagged by PageSpeed on every tested URL.

  • Cross-reference the app list (provided in the table below) against the waterfall output from Action 1
  • For each app, determine: does it load JS globally (all pages) or only on specific page types?
  • Flag apps that load JS globally but are only needed on specific pages — these are candidates for conditional loading
  • Priority apps to investigate first based on known JS weight: Klaviyo, Judge.me, Searchanise, Smile.io / Glow, Octane AI / Visual Quiz Builder, Browsee, Privy, Attrac, Smart Stories, Foursixty
  • Check for duplicate functionality: there are currently 2 quiz apps (Quizify + Octane AI + Visual Quiz Builder + Quiz Kit = 4 quiz apps), 2 search apps (Searchanise + Search & Discovery + Power Tools Filter), 2 affiliate apps (GOAFFPRO + UpPromote), 2 visitor counter apps (GAF + Trafficly), 2 loyalty apps (Smile.io + Glow). Each installed app loads JS even if inactive — confirm which are live and which can be uninstalled
Deliverable: App-by-app table — app name, JS loaded globally (Y/N), estimated JS size, recommendation (keep / make conditional / remove)
Critical Week 1
3

Identify and Quantify Render-Blocking Resources

PageSpeed flagged render-blocking requests on every page but the original audit captured no specifics. Need the actual resource names and estimated delay in milliseconds.

  • Run PageSpeed Insights on homepage and one PDP — expand the "Eliminate render-blocking resources" opportunity and record every listed resource name and its estimated savings in ms
  • Assess font loading strategy — are fonts (Playfair Display, Manrope) loaded via Google Fonts CDN with render-blocking link tags, or via font-display: swap with preconnect?
  • Identify any CSS files loaded in <head> that are not critical for above-the-fold rendering
Deliverable: List of render-blocking resources with file name, type (CSS/JS/font), and estimated ms delay each
Critical Week 1
4

Preload LCP Image — Confirm Implementation Status

Every tested page has an LCP of 3.3–3.9s. The fastest single fix is to preload the LCP image element so the browser discovers it earlier. Assess current state and confirm whether this is already implemented or missing.

  • Check homepage theme code: is there a <link rel="preload" as="image"> for the hero banner?
  • Check PDP theme code: is the first product image preloaded?
  • Check whether all img tags on these pages have explicit width and height attributes — missing attributes cause layout shift and delay LCP discovery
  • If not implemented: provide implementation plan and estimated LCP improvement in ms
Deliverable: Current state confirmation (implemented / not implemented) + estimated LCP improvement if fix applied
Critical Week 1 — quick win
5

Verify WebP Image Delivery via Shopify CDN

Shopify automatically serves WebP if the theme uses the correct Liquid image_url filter syntax. If the theme is using hardcoded image URLs or legacy img_url filters, images are served as JPEG/PNG — adding 30–50% unnecessary payload.

  • Inspect network requests on a PDP — confirm whether product images are served with .webp extension from cdn.shopify.com
  • Check the theme's product image Liquid code for image_url filter usage: | image_url: width: 800, format: 'webp'
  • Check whether srcset is implemented for responsive images (different sizes for mobile vs desktop)
  • Check whether product images uploaded as AI-edited files (UUID-named) behave differently in CDN delivery
Deliverable: Confirmation of WebP status + if not implemented, scope of Liquid template changes required
High Week 1
6

Investigate Non-Composited Animations — 46 Elements on Homepage

The homepage has 46 non-composited animated elements. These run on the main thread and are a direct cause of mobile jank and elevated INP. The Dining collection has 20. PDPs have 4–14.

  • Identify which CSS properties are being animated — look for transitions/animations on top, left, width, height, margin, padding, background, box-shadow
  • These should all be migrated to transform and opacity which are GPU-composited and cost nothing on the main thread
  • Confirm whether animations are theme-native (Expanse OS 2.0) or injected by an app (e.g. Attrac, Smart Stories, Hextom Shipping Bar)
Deliverable: List of animation sources (theme vs app) + scope of CSS changes required
High Week 2
7

Identify the Source of 65 User Timing Marks

Exactly 65 User Timing marks appear on every tested page — homepage, collections, and PDPs. The identical count across all page types is a strong signal of a single globally-injected app or script using the Performance.mark() / Performance.measure() API excessively.

  • In Chrome DevTools Performance panel, record a page load and inspect the "Timings" row — each mark will show with its label
  • The label names will identify which app is responsible
  • Assess whether these marks serve a live function or are debug/analytics artifacts that can be removed from production
  • Suspected candidates based on app list: Browsee (session recording), Klaviyo, qeen.ai CRM, or a custom analytics integration
Deliverable: App/script identified + recommendation (remove marks / make conditional)
High Week 2
8

Investigate TTFB and Server Response Time

Mobile TTFB is 0.7–0.9s. Google's "Good" threshold is under 0.8s — the site is at the boundary. Every millisecond of TTFB delays every other metric downstream including LCP.

  • Test TTFB from a Saudi-based connection (Riyadh / Dammam) vs. a European connection — if the gap is large, CDN edge configuration for the Middle East region may need review
  • Check Shopify Markets configuration — is the Arabic storefront routed to the nearest CDN edge?
  • Review any custom Liquid code that performs complex storefront logic on render (metafield lookups, multi-pass liquid, large collection queries)
Deliverable: TTFB measurement from Saudi IP + identified causes if over 800ms
High Week 2
